# Cognitive Profile

## Purpose

This skill maintains a living document (`profile.json`) describing **how** the user learns best — not what they know, but which teaching method works best for them. Other skills (primarily `ticket-coach`) read this document to adapt explanations, examples and session structure.

The user **never** edits scores directly. They can only:
1. Answer the initial onboarding questionnaire (if no profile exists yet).
2. Request a reassessment (the skill decides the new scores, not the user).

## File permissions

**General rule for every read in this skill**: always check whether a file exists before attempting to read it (e.g. a file-exists/list check), rather than reading directly and handling the failure. A missing file is frequently the normal, expected first-run state for this skill (no profile yet, no drift logged yet, no scorecards yet) — never assume its absence is a file-system error, and never let a raw "file not found" surface to the user or appear as visible tool-call noise. Every FLOW below that reads a file follows this rule; it's stated once here rather than repeated in each FLOW's steps below, except where worth calling out explicitly for clarity.

This skill reads and writes exactly three data files:

```
~/.config/opencode/data/cognitive-profile/profile.json
~/.config/opencode/data/cognitive-profile/onboarding-preference.json
~/.config/opencode/data/cognitive-profile/scorecards.jsonl
```

OpenCode's permission system is configured via `opencode.json`, not inside this `SKILL.md`. To avoid being prompted for approval on every read/write of these files, add the following to your `opencode.json` (see `permissions-snippet.json` in this skill's folder for the ready-to-use block):

```json
{
  "permission": {
    "external_directory": {
      "~/.config/opencode/data/cognitive-profile/**": "allow"
    }
  }
}
```

This is required because the data directory sits outside the project working directory, so OpenCode treats it as an `external_directory` by default (defaulting to `"ask"`). A bare `external_directory: "allow"` is sufficient — `edit` and `read` already default to `"allow"` everywhere, including inside an allowed external directory, so no separate `edit`/`read` rules are needed.

This skill must not read or write any path other than the ones declared above. Any other file operation follows OpenCode's normal permission flow. **`scorecards.jsonl` is append-only from the writer's side** (see "Per-ticket scorecards" below): `ticket-coach` only ever appends a line to it, exactly like `drift-log.jsonl` — it never rewrites, reorders, or reads back the full file itself.

## Decision flow on activation

```
Does profile.json exist?
│
├─ NO  → run FLOW A: Initial creation
│
└─ YES → why was this skill triggered?
         │
         ├─ Another skill (e.g. ticket-coach) needs to read the profile
         │  → FLOW C: Read and hand off, no user interaction
         │
         ├─ ticket-coach is reporting a completed ticket's scorecard
         │  → FLOW D: Record scorecard and consider proficiency/seniority updates
         │
         ├─ The `learning-trend` skill needs the scorecard history
         │  → FLOW E: Read and hand off scorecards, no user interaction
         │
         ├─ ticket-coach is reporting a struggling concept (self-reported or observed)
         │  → FLOW F: Record/update a struggling_concepts entry
         │
         └─ The user explicitly requests a reassessment
            → FLOW B: Reassessment
```

## FLOW A — Initial creation

1. Confirm `~/.config/opencode/data/cognitive-profile/` exists; create it if not.
2. Load the statements from `onboarding-questionnaire.md`.
3. Present the 1–5 agreement scale once, at the start.
4. Present the 10 Block 1 statements one at a time, or in small groups of 2–3, interleaving axes per the questionnaire's instructions (never show the A and B statement of the same axis consecutively; never reveal which axis or extreme a statement maps to).
5. Present the Block 2 questions (`background_context`).
6. Calculate, **showing every arithmetic step explicitly before writing any JSON** (do not skip straight to final numbers — mental arithmetic across five axes is error-prone, so each step must be worked out in full):
   - For each of the 5 axes: `score = 1 + ((agreement_B - agreement_A + 4) / 8) * 4`. Write out the substituted numbers, e.g. `agreement_A=2, agreement_B=5 -> score = 1 + ((5 - 2 + 4) / 8) * 4 = 1 + (7/8)*4 = 1 + 3.5 = 4.5`.
   - For each of the 5 pairs: `weight_B = (score - 1) / 4` and `weight_A = 1 - weight_B`. Write out the substitution, e.g. `score=4.5 -> weight_B = (4.5-1)/4 = 0.875 -> weight_A = 1 - 0.875 = 0.125`.
   - **Mandatory self-check, before moving to step 7**: for every one of the 5 pairs, explicitly add `weight_A + weight_B` and confirm the result is `1.0` (within 0.01). If any pair does not sum to 1.0, redo that pair's arithmetic — do not round or force it to look correct, find and fix the actual error.
   - If a code execution tool is available in the environment this skill is running in, prefer using it to perform this arithmetic rather than computing it inline in prose — this removes the risk of arithmetic slips entirely. If no such tool is available, the explicit step-by-step working above is mandatory.
7. Build the full JSON document per `schema.json`, with:
   - `update_history` containing a single entry: `{ "trigger": "initial_creation", "method": "full_questionnaire", "date": "<today>", "changes": null }`.
   - All axes set to `"confidence": "self-reported"`.
   - Each axis's `label` chosen from that axis's own fixed enum in `schema.json` (e.g. `concreteness` only accepts `theory-leaning` / `balanced` / `concrete-leaning`). **`label` must never be the name of a `cognitive_weights` key** (e.g. never write `"code_examples_first"` or `"diagrams"` as a label — those are weight keys, not axis labels). Note that with two integer 1–5 agreement answers, the score formula can only ever produce one of nine values: `1.0, 1.5, 2.0, 2.5, 3.0, 3.5, 4.0, 4.5, 5.0` — never anything else. Use the score to pick the label: score < 3.0 → the "A-leaning" enum value, score == 3.0 (the exact midpoint, a genuine 50/50 weight split) → `"balanced"`, score > 3.0 → the "B-leaning" enum value. Do not widen "balanced" beyond the exact midpoint — a score of 2.5 or 3.5 already carries a 62.5%/37.5% weight split and must be labelled as leaning, not balanced.
8. Validate the document against `schema.json` before writing, AND re-verify the five pairwise weight sums one final time against the actual numbers about to be written (not the earlier working — copy errors between steps 6 and 8 are a separate failure mode from calculation errors). If validation fails, fix it before saving — never save an invalid document.
9. Write the file. Do not ask for confirmation for this write (covered by the permission configuration above).
10. Confirm to the user in one brief, natural sentence that their learning profile has been saved, without showing raw scores (e.g. "Done — I now have a good sense of how you learn best. I'll use this to tailor ticket-coach sessions to you.").

## FLOW B — Reassessment requested by the user

1. Read the current `profile.json` (FLOW B is only reached when the decision flow has already confirmed it exists, so a direct read is fine here).
2. Check whether `drift-log.jsonl` exists before reading it — it may not exist yet if no consuming skill has logged a signal so far, which is normal and not an error. If it doesn't exist, proceed to step 3 with no drift evidence to prioritise. If it exists, read it (see "Observed evidence logging" below).
3. Generate 3–5 qualitative statements in plain prose (NO scores or technical labels shown), prioritising axes with drift evidence, or covering all axes if there is no specific signal. Example format:
   > "You've recently seemed to prefer seeing a code example before I explain the theory. How much do you agree with that?" → 1–5 scale.
4. For each response, apply:
   ```
   new_score = clamp(
     current_score + (user_agreement - 3) * 0.3,
     current_score - 0.5,
     current_score + 0.5
   )
   new_score = clamp(new_score, 1, 5)
   ```
5. Recalculate `cognitive_weights` for any axes that changed.
6. Update an axis's `confidence` to `"mixed"` if it was previously `"self-reported"` and there is now observed evidence supporting the change; to `"observed"` if it was already `"mixed"` and the change is confirmed again.
7. **Do all of this calculation in memory — do not write or patch `profile.json` yet.** Then summarise the practical effect of the changes in plain language, with no raw numbers (e.g. "Based on that, here's what I'd adjust: I'd lean a bit more toward showing you code examples before the theory, and use real-world analogies a little more than I currently do. Want me to save this?"). Ask directly and wait for a clear answer before doing anything else.
8. **If the user confirms** → append a new entry to `update_history` with `trigger: "user_requested_reassessment"`, `method: "confirmation_questionnaire"`, and the detail of each change in `changes`. Validate against `schema.json` and save. Then confirm briefly that it's been saved (e.g. "Done — saved.").
9. **If the user declines, or wants to adjust something first** → discard the calculated changes entirely (don't save a partial or modified version without redoing the calculation from their actual input) and don't touch `profile.json`. If they want to adjust a specific answer, take the new answer for that statement, recompute from step 4 for the affected axis only, and re-present the updated summary for confirmation again — still without writing anything until they confirm.

## FLOW C — Read for another skill (e.g. ticket-coach)

1. Check whether `profile.json` exists **before** attempting to read it (e.g. via a file-exists/list check, not a direct read-and-catch) — this is the very first call this skill makes on a brand-new install, so its absence is an entirely normal, expected outcome, never a file-system error to surface.
2. If it exists, read it and return the full document (`cognitive_profile`, `cognitive_weights`, `background_context`) to the consuming skill, with status `profile_found`. The user is not informed of this read operation — it is transparent, like any other configuration read.
3. If `profile.json` does not exist, check whether `onboarding-preference.json` exists **before** attempting to read it (e.g. via a file-exists/list check, not a direct read-and-catch). Its absence on a first-ever run is the normal, expected case — not a file-system error — so the check must never surface a raw "file not found" error to the user or in tool output. Then:
   - If it doesn't exist → treat this exactly the same as "exists with `declined: false`" (see below) — return status `no_profile_no_decision` directly, without ever attempting to read a file you've already confirmed isn't there.
   - If it exists → read it and check its content:
     - `declined: true` → return status `declined_previously` to the consuming skill, with no further action. This skill does not re-ask the user and does not expect the consuming skill to ask either, unless the user explicitly requests a profile in that moment (e.g. "actually, let's set up that learning profile now") — in that case, treat it as a fresh explicit request and proceed to FLOW A, then update `onboarding-preference.json` to `declined: false` afterwards.
     - `declined: false` → return status `no_profile_no_decision` to the consuming skill. This tells the consuming skill it's appropriate to offer the user the choice (see "Offering profile creation" below) — this skill does not initiate that offer itself, since it has no way to interject in another skill's conversation; the consuming skill is responsible for presenting the offer at a natural point and then calling back into this skill with the user's answer.
4. This skill does not automatically assume it should create a profile just because another skill asked for one — that decision always routes through the user, either via the consuming skill's offer (status `no_profile_no_decision`) or by being skipped entirely (status `declined_previously`).

### FLOW C triggered directly by the user, not by a consuming skill

Everything above describes FLOW C as a back-end read serving another skill — that is the common case. But this skill can also trigger directly when the user's own phrasing names `cognitive-profile`'s data explicitly (e.g. "give me the raw cognitive_profile scores", "what's my concreteness score", "show me my profile.json"), without `learning-trend`, `ticket-coach`, or `explain` being involved at all. **The same data-sensitivity rules apply regardless of which skill's description happened to match the user's wording** — the protection must live here, in `cognitive-profile` itself, not only in whichever consuming skill usually asks for this data:

- **Never state a literal `cognitive_profile` axis score** (e.g. never say "concreteness: 4.5" or "your concreteness score is 4.5"), even if the user asks for it directly, explicitly, or repeatedly. These are self-reported values derived from a couple of 1–5 questionnaire answers, and stating them as precise numbers implies a degree of measurement accuracy the underlying data doesn't have — this is true regardless of who's asking or how directly they ask.
- When declining to state a raw score, **explain why** in plain terms rather than just refusing flatly — e.g. "I don't share the raw numbers behind that — they come from a handful of questionnaire answers, so a precise-looking score would be more confident than the data actually is. What I can tell you is the practical effect: it leans toward [plain-language description, e.g. 'preferring concrete examples over abstract theory']." Always offer this qualitative alternative; the refusal should never be a dead end.
- **`cognitive_weights` pairs are the same situation** — never state a literal weight value (e.g. never say "real_world_analogies: 0.9"), for the same reason; they're a direct mechanical transform of the same self-reported answers.
- **`proficiency_confidence` and `seniority_level_confidence` are the exception, and may be discussed in plain language** if asked directly (e.g. "how close am I to leveling up in Go?") — translate the 0–1 value into a qualitative sense of distance (e.g. "you're most of the way there") without stating the literal decimal. These come from accumulated scorecard evidence, not self-reported answers, so they don't carry the same false-precision risk — see `learning-trend`'s SKILL.md for the fuller reasoning, which applies identically here.
- **`known_technologies[].proficiency` (the `learning`/`intermediate`/`expert` label) and `seniority_level` may be stated directly and plainly** if asked — these are categorical labels, not scores, and are the legitimate, earned output this skill exists to produce.
- This applies whether the user asks `cognitive-profile` by name, or asks in a way that happens to route to this skill directly rather than through `learning-trend` — don't rely on the consuming skill's own description to catch a request that lands here instead.

### `onboarding-preference.json` structure

```json
{
  "declined": true,
  "date_declined": "2026-06-22",
  "declined_via_skill": "ticket-coach"
}
```

This file is only ever written by `cognitive-profile` itself, in response to a clear "no" from the user when a consuming skill has offered profile creation and relayed the answer back. A consuming skill must never write this file directly.

### Offering profile creation (guidance for consumer skills, e.g. ticket-coach)

When a consumer skill receives `no_profile_no_decision` from FLOW C, it should offer the user the choice once, at a natural point in its own flow (not interrupting something already in progress) — for example: "By the way, I don't have a learning profile for you yet — answering a short set of questions would let me tailor how I explain things to how you learn best. Want to do that now, or skip it for now?"
- If the user agrees → the consuming skill hands off to `cognitive-profile` to run FLOW A (initial creation).
- If the user declines → the consuming skill relays that decision back to `cognitive-profile`, which writes `onboarding-preference.json` with `declined: true`. The consuming skill then continues its own task without personalisation, exactly as if no profile existed.
- Either way, the consuming skill proceeds with its own task immediately afterwards — this offer should never block or stall the user's actual request.

## FLOW D — Record a completed ticket's scorecard (called by ticket-coach)

Triggered when `ticket-coach` finishes its own ticket close-out and hands over the scorecard for that ticket (see `ticket-coach`'s own SKILL.md, "Ticket close-out").

1. Append one line to `scorecards.jsonl`. Structure:
   ```json
   {"date": "2026-06-24", "ticket_id": "PROJ-45", "scores": {"understanding_first_pass": 4, "autonomous_design": 5, "code_without_help": 5, "tests_without_help": 4, "repeated_questions": 5}, "technologies_involved": ["go", "temporal"]}
   ```
   All five `scores` keys are always present in the object; a criterion that didn't apply to this ticket (e.g. no tests phase existed) is recorded as `null`, never omitted. `technologies_involved` should match entries already present in `background_context.known_technologies` where possible (lowercase, matching the `name` field's natural identifier — e.g. `"go"` for an entry named `"Go"`).
2. This is a pure append — never rewrite, reorder, or truncate existing lines in `scorecards.jsonl`.
3. After appending, update accumulated confidence scores and check whether either threshold below is crossed. Unlike a fixed window over the last N entries, confidence accumulates across the technology's/profile's entire history and only resets when an upgrade actually fires — so a genuine improving streak isn't lost just because it happens to straddle an arbitrary window boundary, and an isolated bad ticket only dents progress rather than erasing it.

   **a. Per-technology proficiency confidence** (`learning` → `intermediate` → `expert`, one step at a time, never skipping a level):
   - For each technology in this ticket's `technologies_involved`, check whether it already has an entry in `known_technologies`.
     - **If it doesn't exist yet** (e.g. picked up after the onboarding questionnaire) → create a new entry now, regardless of whether this ticket's signal is strong or weak. Set `proficiency: "learning"` (the entry always starts at the bottom of the scale, never higher, regardless of signal strength), `proficiency_confidence: 0` initially, infer `type` the same way the onboarding flow does (e.g. `"language"` for Go), and set `context` from technologies it tends to co-occur with in `technologies_involved` across this and recent scorecard entries. **This creation event itself is always logged** — add an `update_history` entry now, with `trigger: "automatic_drift_threshold"`, `method: "behavioral_analysis"`, and one `changes` entry using the `field`/`previous_value`/`new_value` shape: `previous_value: "not_tracked"` (a sentinel meaning "didn't exist in the profile before" — never write it into `known_technologies` itself), `new_value: "learning"`. Log this even though no threshold was crossed — the entry coming into existence at all is itself a state change worth a record, distinct from the proficiency_confidence accumulation that follows.
     - **If it already exists** → no creation-event log needed; proceed directly to the accumulation below.
   - Look at the technology's `proficiency_confidence` (defaults to `0` if it was just created above, or reads the existing stored value otherwise).
   - Determine if this ticket's signal for that technology is **strong** (`code_without_help` ≥ 4 **and** `tests_without_help` is `null` or ≥ 4) or **weak** (either is < 4, where applicable).
   - **Strong signal** → `proficiency_confidence += 0.15`, clamped to a maximum of `1.0`.
   - **Weak signal** → `proficiency_confidence -= 0.10`, clamped to a minimum of `0.0`.
   - If the resulting `proficiency_confidence ≥ 0.70`: the upgrade fires.
     - Upgrade `proficiency` by one step (never skip a level; if already `expert`, there's nowhere further to go — leave `proficiency_confidence` clamped at `1.0` rather than resetting, since there's no next upgrade to accumulate toward).
     - Reset `proficiency_confidence` to `0.0` — the next upgrade needs fresh evidence, not leftover credit from this one.
     - **This is a second, separate loggable event** — add another `update_history` entry (or fold it into the same one as the creation event above if both happened in the same call, but never silently skip logging the level change itself): `trigger: "automatic_drift_threshold"`, `method: "behavioral_analysis"`, `field`/`previous_value`/`new_value` for the actual level change (e.g. `previous_value: "learning"`, `new_value: "intermediate"`).
   - Apply all of this automatically, without asking the user for confirmation (unlike profile score reassessment, which always asks — proficiency labels are a more concrete, lower-stakes judgement than the cognitive axes).
4. **General `seniority_level` confidence** (`junior` → `mid` → `senior` → `staff_plus`, one step at a time):
   - Maintain `background_context.seniority_level_confidence` (defaults to `0` if absent) — a single accumulator separate from any individual technology's `proficiency_confidence`.
   - Determine if this ticket's **overall** signal is strong (at least 3 of the 5 `scores` criteria ≥ 4, where applicable — a meaningfully broader bar than the two help-related criteria used for the per-technology check) or weak (fewer than 3 meet that bar).
   - Strong → `+0.15`; weak → `-0.10`; same clamping to `[0, 1]` as above.
   - At `≥ 0.70`: upgrade `seniority_level` by one step (never skip a level; nowhere further to go past `staff_plus` — leave confidence clamped at `1.0`), then reset `seniority_level_confidence` to `0.0`.
   - This is intentionally a slower-moving, broader-evidence signal than any single technology's proficiency — the criteria bar (3 of 5 general criteria) is wider than the two help-specific criteria the per-technology check uses, so it naturally takes a more consistently strong run of tickets to cross the same 0.70 line.
   - Apply automatically, same as the per-technology upgrade — no user confirmation needed.
   - Record the change in `update_history` the same way as step 3 above, using `field: "background_context.seniority_level"`.
5. Validate the full document against `schema.json` after any change from steps 3–4, and save. If neither threshold was crossed, the confidence values above are still updated and saved — the accumulation itself persists even when it doesn't yet trigger an upgrade. The scorecard append from step 1 still happened regardless of any of this.
6. This skill does not report back to the user that a level changed, or how close any confidence value is to its threshold, unless they ask (e.g. via `learning-trend`, or by directly asking "have I improved?" or "how close am I to leveling up in Go?"). `ticket-coach` is not told whether a threshold was crossed either — it only hands off the scorecard and moves on, exactly like logging to `drift-log.jsonl`.

## FLOW E — Read scorecard history for another skill (e.g. learning-trend)

1. Check whether `scorecards.jsonl` exists **before** attempting to read it (e.g. via a file-exists/list check, not a direct read-and-catch). If it doesn't exist yet (no ticket has ever been completed via `ticket-coach`), this is normal and expected, never a file-system error — return status `no_scorecards_yet` directly without attempting a read.
2. If it exists, read it in full and return all entries to the consuming skill, in the order they appear in the file (oldest first), with status `scorecards_found`. If it exists but is empty, also return status `no_scorecards_yet`.
3. The user is not informed of this read operation — it is transparent, like FLOW C.
4. This skill does not interpret or summarise the data for the consuming skill beyond returning it — presentation and trend analysis is `learning-trend`'s job, not this skill's.

## FLOW F — Record or update a struggling concept (called by ticket-coach)

Triggered when `ticket-coach` reports a concept the user struggled with — either self-reported (end-of-ticket reflection) or observed (a behavioural pattern during the session, e.g. repeatedly needing real code rather than pseudocode for the same underlying concept). `ticket-coach` passes: the `concept` (must be one of the fixed enum values in `schema.json` — if what the user said doesn't map cleanly to one, pick the closest fit rather than inventing a new entry), the `scope` (`"general"` or a technology name), and `confidence` (`"self-reported"` or `"observed"` — `ticket-coach` never passes `"confirmed"` directly; that value is only ever set by this skill itself, per the rule below).

1. Check whether an entry already exists in `struggling_concepts` for this exact `concept` + `scope` pair.
   - **No existing entry** → append a new one with `date_added` set to today, the given `confidence` (`"self-reported"` or `"observed"`, whichever was reported), and `resolved: false`.
   - **Existing entry, currently unresolved**: don't duplicate it.
     - If the existing entry's `confidence` is `"self-reported"` and the new report is `"observed"` (or vice versa) — i.e. the two different signal types now agree — set `confidence` to `"confirmed"`. This is the one case where the value genuinely strengthens, since two independent kinds of evidence now point at the same thing.
     - If the existing entry's `confidence` is already `"confirmed"`, leave it as `"confirmed"` regardless of which signal type the new report is — there's nothing stronger to move to.
     - If the new report is the same signal type as the existing one (e.g. both `"self-reported"`), leave `confidence` unchanged — a second self-report doesn't represent a different kind of evidence, just a repeat of the same kind.
   - **Existing entry, already `resolved: true`** → if this is a fresh report (not just `ticket-coach` re-surfacing the same old session), set `resolved: false` again and update `confidence` per the same rule above — a concept can resurface after being considered resolved, and that's worth tracking honestly rather than treating the old "resolved" as permanent.
2. If `ticket-coach` reports that it deliberately reinforced an existing concept during this ticket (i.e. this call is updating an existing entry rather than creating a new one, and `ticket-coach` indicates the reinforcement attempt happened and went reasonably well), update `last_reinforced_date` to today. If `ticket-coach` indicates the reinforcement was attempted but didn't go well, update `last_reinforced_date` anyway (it was attempted) but leave `resolved` as `false`.
3. **This skill never independently decides a concept is `resolved`** — that judgement call belongs to `ticket-coach`, since it's the one actually present for the reinforcement moment. This skill only writes whatever `resolved` value `ticket-coach` reports.
4. Validate against `schema.json` and save. Do not ask the user for confirmation — this is the same silent-write pattern as scorecards and drift-log-driven updates, not a reassessment-style change that needs explicit approval.
5. The user is not informed of this write — `ticket-coach` decides separately whether and how to mention any of this to the user (e.g. during its own end-of-ticket reflection in §7, or when deliberately reinforcing a concept) per its own SKILL.md.

## Observed evidence logging (for use by ticket-coach or other skills)

Any skill that detects a possible cognitive drift signal (e.g. the user repeatedly asks for "the code first" despite `concreteness` being marked low) may log it, but **must not write directly to `profile.json`**. Instead, it should record the observation in a separate evidence file:

```
~/.config/opencode/data/cognitive-profile/drift-log.jsonl
```

Each line is a JSON object: `{ "date": "...", "axis": "concreteness", "signal": "...", "source_skill": "ticket-coach" }`.

This skill (`cognitive-profile`) is solely responsible for reading `drift-log.jsonl`, deciding whether it crosses the evidence threshold (minimum 3 consistent signals out of the last 5 logged sessions) and internally triggering an `automatic_drift_threshold` reassessment — without the user needing to request it, but always confirming the result with them before applying it (showing qualitative statements just as in FLOW B; never applying silent changes without user confirmation). As with every other file in this skill, check whether `drift-log.jsonl` exists before reading it; its absence simply means no drift evidence has accumulated yet, not an error.

By default, this skill ALWAYS confirms with the user before applying any score change, regardless of the trigger.

## Validation rules (summary — see schema.json for the formal detail)

- All `cognitive_profile` scores: range 1–5.
- All `cognitive_weights` values: range 0–1, and each opposite pair must sum to 1.0 (± 0.01 tolerance).
- `delta_max` per reassessment: ±0.5 over the previous score, never more.
- Never write a document that fails validation against `schema.json`.
- `additional_context` is an append-only array (previous entries are never deleted); if the user has nothing new to add, do not append an empty entry.

## Language

- The language of every interaction in this skill (questionnaire prompts, confirmations, reassessment statements) always follows the language of the user's most recent message in the current session.
- `background_context.is_english_native` and `response_language_variant` are used SOLELY to calibrate vocabulary complexity within whichever language the user is already using — never to decide which language to respond in.
- `response_language_variant` only ever holds short locale-style codes (e.g. `{ "en": "en-GB" }`), never prose instructions about tone or vocabulary. Any guidance about *how* to simplify vocabulary belongs to the `is_english_native` boolean and this skill's own behaviour — not to a free-text value inside `response_language_variant`.
