# Cognitive Profile

## Purpose

This skill maintains a living document (`profile.json`) describing **how** the user learns best — not what they know, but which teaching method works best for them. Other skills (primarily `ticket-coach`) read this document to adapt explanations, examples and session structure.

The user **never** edits scores directly. They can only:
1. Answer the initial onboarding questionnaire (if no profile exists yet).
2. Request a reassessment (the skill decides the new scores, not the user).

## File permissions

This skill reads and writes exactly three data files:

```
~/.config/opencode/data/cognitive-profile/profile.json
~/.config/opencode/data/cognitive-profile/onboarding-preference.json
~/.config/opencode/data/cognitive-profile/scorecards.jsonl
```

OpenCode's permission system is configured via `opencode.json`, not inside this `SKILL.md`. To avoid being prompted for approval on every read/write of these files, add the following to your `opencode.json` (see `permissions-snippet.json` in this skill's folder for the ready-to-use block):

```json
{
  "permission": {
    "external_directory": {
      "~/.config/opencode/data/cognitive-profile/**": "allow"
    }
  }
}
```

This is required because the data directory sits outside the project working directory, so OpenCode treats it as an `external_directory` by default (defaulting to `"ask"`). A bare `external_directory: "allow"` is sufficient — `edit` and `read` already default to `"allow"` everywhere, including inside an allowed external directory, so no separate `edit`/`read` rules are needed.

This skill must not read or write any path other than the ones declared above. Any other file operation follows OpenCode's normal permission flow. **`scorecards.jsonl` is append-only from the writer's side** (see "Per-ticket scorecards" below): `ticket-coach` only ever appends a line to it, exactly like `drift-log.jsonl` — it never rewrites, reorders, or reads back the full file itself.

## Relevant paths

| File | Path | Purpose |
|---|---|---|
| User profile (data) | `~/.config/opencode/data/cognitive-profile/profile.json` | Mutable state, read/written by this skill |
| Onboarding preference | `~/.config/opencode/data/cognitive-profile/onboarding-preference.json` | Records whether the user has declined profile creation, so consumer skills (e.g. ticket-coach) stop asking until the user explicitly requests a profile |
| JSON Schema (validation) | `~/.config/opencode/skills/cognitive-profile/schema.json` | Formal contract for the document |
| Onboarding questionnaire | `~/.config/opencode/skills/cognitive-profile/onboarding-questionnaire.md` | Onboarding statements |
| Drift evidence log | `~/.config/opencode/data/cognitive-profile/drift-log.jsonl` | Behavioural signals logged by consumer skills (e.g. ticket-coach) |
| Per-ticket scorecards | `~/.config/opencode/data/cognitive-profile/scorecards.jsonl` | One line per completed ticket, appended by `ticket-coach`; read and interpreted only by this skill (e.g. by `learning-trend`, via FLOW E) |

## Decision flow on activation

```
Does profile.json exist?
│
├─ NO  → run FLOW A: Initial creation
│
└─ YES → why was this skill triggered?
         │
         ├─ Another skill (e.g. ticket-coach) needs to read the profile
         │  → FLOW C: Read and hand off, no user interaction
         │
         ├─ ticket-coach is reporting a completed ticket's scorecard
         │  → FLOW D: Record scorecard and consider proficiency/seniority updates
         │
         ├─ The `learning-trend` skill needs the scorecard history
         │  → FLOW E: Read and hand off scorecards, no user interaction
         │
         └─ The user explicitly requests a reassessment
            → FLOW B: Reassessment
```

## FLOW A — Initial creation

1. Confirm `~/.config/opencode/data/cognitive-profile/` exists; create it if not.
2. Load the statements from `onboarding-questionnaire.md`.
3. Present the 1–5 agreement scale once, at the start.
4. Present the 10 Block 1 statements one at a time, or in small groups of 2–3, interleaving axes per the questionnaire's instructions (never show the A and B statement of the same axis consecutively; never reveal which axis or extreme a statement maps to).
5. Present the Block 2 questions (`background_context`).
6. Calculate, **showing every arithmetic step explicitly before writing any JSON** (do not skip straight to final numbers — mental arithmetic across five axes is error-prone, so each step must be worked out in full):
   - For each of the 5 axes: `score = 1 + ((agreement_B - agreement_A + 4) / 8) * 4`. Write out the substituted numbers, e.g. `agreement_A=2, agreement_B=5 -> score = 1 + ((5 - 2 + 4) / 8) * 4 = 1 + (7/8)*4 = 1 + 3.5 = 4.5`.
   - For each of the 5 pairs: `weight_B = (score - 1) / 4` and `weight_A = 1 - weight_B`. Write out the substitution, e.g. `score=4.5 -> weight_B = (4.5-1)/4 = 0.875 -> weight_A = 1 - 0.875 = 0.125`.
   - **Mandatory self-check, before moving to step 7**: for every one of the 5 pairs, explicitly add `weight_A + weight_B` and confirm the result is `1.0` (within 0.01). If any pair does not sum to 1.0, redo that pair's arithmetic — do not round or force it to look correct, find and fix the actual error.
   - If a code execution tool is available in the environment this skill is running in, prefer using it to perform this arithmetic rather than computing it inline in prose — this removes the risk of arithmetic slips entirely. If no such tool is available, the explicit step-by-step working above is mandatory.
7. Build the full JSON document per `schema.json`, with:
   - `update_history` containing a single entry: `{ "trigger": "initial_creation", "method": "full_questionnaire", "date": "<today>", "changes": null }`.
   - All axes set to `"confidence": "self-reported"`.
   - Each axis's `label` chosen from that axis's own fixed enum in `schema.json` (e.g. `concreteness` only accepts `theory-leaning` / `balanced` / `concrete-leaning`). **`label` must never be the name of a `cognitive_weights` key** (e.g. never write `"code_examples_first"` or `"diagrams"` as a label — those are weight keys, not axis labels). Note that with two integer 1–5 agreement answers, the score formula can only ever produce one of nine values: `1.0, 1.5, 2.0, 2.5, 3.0, 3.5, 4.0, 4.5, 5.0` — never anything else. Use the score to pick the label: score < 3.0 → the "A-leaning" enum value, score == 3.0 (the exact midpoint, a genuine 50/50 weight split) → `"balanced"`, score > 3.0 → the "B-leaning" enum value. Do not widen "balanced" beyond the exact midpoint — a score of 2.5 or 3.5 already carries a 62.5%/37.5% weight split and must be labelled as leaning, not balanced.
8. Validate the document against `schema.json` before writing, AND re-verify the five pairwise weight sums one final time against the actual numbers about to be written (not the earlier working — copy errors between steps 6 and 8 are a separate failure mode from calculation errors). If validation fails, fix it before saving — never save an invalid document.
9. Write the file. Do not ask for confirmation for this write (covered by the permission configuration above).
10. Confirm to the user in one brief, natural sentence that their learning profile has been saved, without showing raw scores (e.g. "Done — I now have a good sense of how you learn best. I'll use this to tailor ticket-coach sessions to you.").

## FLOW B — Reassessment requested by the user

1. Read the current `profile.json`.
2. Check for accumulated entries in `drift-log.jsonl` (see "Observed evidence logging" below).
3. Generate 3–5 qualitative statements in plain prose (NO scores or technical labels shown), prioritising axes with drift evidence, or covering all axes if there is no specific signal. Example format:
   > "You've recently seemed to prefer seeing a code example before I explain the theory. How much do you agree with that?" → 1–5 scale.
4. For each response, apply:
   ```
   new_score = clamp(
     current_score + (user_agreement - 3) * 0.3,
     current_score - 0.5,
     current_score + 0.5
   )
   new_score = clamp(new_score, 1, 5)
   ```
5. Recalculate `cognitive_weights` for any axes that changed.
6. Update an axis's `confidence` to `"mixed"` if it was previously `"self-reported"` and there is now observed evidence supporting the change; to `"observed"` if it was already `"mixed"` and the change is confirmed again.
7. Append a new entry to `update_history` with `trigger: "user_requested_reassessment"`, `method: "confirmation_questionnaire"`, and the detail of each change in `changes`.
8. Validate against `schema.json` and save. Do not ask for confirmation to write.
9. Confirm to the user in plain language, without showing numbers (e.g. "Right, I've adjusted your profile based on that — you should notice it in how I explain things from now on.").

## FLOW C — Read for another skill (e.g. ticket-coach)

1. Read `profile.json`.
2. If it exists, return the full document (`cognitive_profile`, `cognitive_weights`, `background_context`) to the consuming skill, with status `profile_found`. The user is not informed of this read operation — it is transparent, like any other configuration read.
3. If `profile.json` does not exist, read `onboarding-preference.json` (see its structure below):
   - If it exists and `declined: true` → return status `declined_previously` to the consuming skill, with no further action. This skill does not re-ask the user and does not expect the consuming skill to ask either, unless the user explicitly requests a profile in that moment (e.g. "actually, let's set up that learning profile now") — in that case, treat it as a fresh explicit request and proceed to FLOW A, then update `onboarding-preference.json` to `declined: false` afterwards.
   - If it does not exist, or exists with `declined: false` → return status `no_profile_no_decision` to the consuming skill. This tells the consuming skill it's appropriate to offer the user the choice (see "Offering profile creation" below) — this skill does not initiate that offer itself, since it has no way to interject in another skill's conversation; the consuming skill is responsible for presenting the offer at a natural point and then calling back into this skill with the user's answer.
4. This skill does not automatically assume it should create a profile just because another skill asked for one — that decision always routes through the user, either via the consuming skill's offer (status `no_profile_no_decision`) or by being skipped entirely (status `declined_previously`).

### `onboarding-preference.json` structure

```json
{
  "declined": true,
  "date_declined": "2026-06-22",
  "declined_via_skill": "ticket-coach"
}
```

This file is only ever written by `cognitive-profile` itself, in response to a clear "no" from the user when a consuming skill has offered profile creation and relayed the answer back. A consuming skill must never write this file directly.

### Offering profile creation (guidance for consumer skills, e.g. ticket-coach)

When a consumer skill receives `no_profile_no_decision` from FLOW C, it should offer the user the choice once, at a natural point in its own flow (not interrupting something already in progress) — for example: "By the way, I don't have a learning profile for you yet — answering a short set of questions would let me tailor how I explain things to how you learn best. Want to do that now, or skip it for now?"
- If the user agrees → the consuming skill hands off to `cognitive-profile` to run FLOW A (initial creation).
- If the user declines → the consuming skill relays that decision back to `cognitive-profile`, which writes `onboarding-preference.json` with `declined: true`. The consuming skill then continues its own task without personalisation, exactly as if no profile existed.
- Either way, the consuming skill proceeds with its own task immediately afterwards — this offer should never block or stall the user's actual request.

## FLOW D — Record a completed ticket's scorecard (called by ticket-coach)

Triggered when `ticket-coach` finishes its own ticket close-out and hands over the scorecard for that ticket (see `ticket-coach`'s own SKILL.md, "Ticket close-out").

1. Append one line to `scorecards.jsonl`. Structure:
   ```json
   {"date": "2026-06-24", "ticket_id": "PROJ-45", "scores": {"understanding_first_pass": 4, "autonomous_design": 5, "code_without_help": 5, "tests_without_help": 4, "repeated_questions": 5}, "technologies_involved": ["go", "temporal"]}
   ```
   All five `scores` keys are always present in the object; a criterion that didn't apply to this ticket (e.g. no tests phase existed) is recorded as `null`, never omitted. `technologies_involved` should match entries already present in `background_context.known_technologies` where possible (lowercase, matching the `name` field's natural identifier — e.g. `"go"` for an entry named `"Go"`).
2. This is a pure append — never rewrite, reorder, or truncate existing lines in `scorecards.jsonl`.
3. After appending, check whether this new entry plus recent history crosses either of the two thresholds below. Both checks only ever look at the **last 5 entries per technology** (for the proficiency check) or **last 5 entries overall** (for the seniority check) — older history is not considered, so a stale period of struggle doesn't permanently suppress a later genuine improvement.

   **a. Per-technology proficiency upgrade** (`learning` → `intermediate` → `expert`, one step at a time, never skipping a level):
   - For a given technology in `technologies_involved`, look at the last 5 scorecard entries that mention it.
   - If at least 3 of those 5 have `code_without_help` ≥ 4 **and** (`tests_without_help` is `null` or ≥ 4) — i.e. consistently resolving with little or no help, where that criterion applied — act according to whether this technology already has an entry in `known_technologies`:
     - **If it already exists** → upgrade its `proficiency` by one step, exactly as before.
     - **If it doesn't exist yet** (e.g. the user picked it up after the onboarding questionnaire, so it was never mentioned) → create a new entry instead of trying to "upgrade" nothing. Set `proficiency: "learning"` regardless of how strong the signal is — the entry is new to the profile, so start at the bottom of the scale rather than assuming a higher starting point the user never confirmed. Infer `type` the same way the onboarding flow does (e.g. `"language"` for Go), and set `context` from the technologies it tends to co-occur with in `technologies_involved` across the matching scorecard entries (e.g. `"Temporal, NATS"`). Don't immediately upgrade this same new entry again in the same FLOW D run, even if the evidence would technically support it — let it surface as `learning` first and pick up the next upgrade on a future ticket, so there's at least one visible step rather than a same-session jump.
   - Apply this automatically, without asking the user for confirmation (unlike profile score reassessment, which always asks — proficiency labels are a more concrete, lower-stakes judgement than the cognitive axes). This applies equally to creating a new entry as to upgrading an existing one — both are silent, like everything else in this flow.
   - Never upgrade more than one step per check, even if the signal is very strong — let it compound naturally across future tickets instead of jumping straight to `expert`.
   - Add an `update_history` entry with `trigger: "automatic_drift_threshold"`, `method: "behavioral_analysis"`, and one `changes` entry using the `field`/`previous_value`/`new_value` shape (not the `axis`/score shape, which is reserved for cognitive axis deltas) — e.g. `{"field": "known_technologies.go.proficiency", "previous_value": "learning", "new_value": "intermediate", "evidence_summary": "4 of last 5 Go tickets resolved with little to no code help"}`. For a newly created entry, use `previous_value: "not_tracked"` (this is a sentinel for "didn't exist in the profile before", not a real `proficiency` enum value, so don't write it anywhere in `known_technologies` itself) and `new_value: "learning"` — e.g. `{"field": "known_technologies.go.proficiency", "previous_value": "not_tracked", "new_value": "learning", "evidence_summary": "4 of last 5 tickets involving Go resolved with little to no code help; Go wasn't in the profile before this"}`.
4. **General `seniority_level` upgrade** (`junior` → `mid` → `senior` → `staff_plus`, one step at a time):
   - This is a second-order signal: it only fires when **multiple distinct technologies have had a proficiency upgrade within the last 5 scorecard entries overall**, AND the same last-5-overall window shows consistently high scores across at least 3 of the 5 score criteria (≥ 4, where applicable) — not just the two help-related ones used for the per-technology check.
   - This is a meaningfully higher bar than the per-technology check by design — general seniority should move slower and on broader evidence than a single technology's proficiency.
   - Apply automatically, same as the per-technology upgrade — no user confirmation needed.
   - Never upgrade more than one step per check.
   - Record the change in `update_history` the same way as step 3 above, using `field: "background_context.seniority_level"`.
5. Validate the full document against `schema.json` after any change from steps 3–4, and save. If nothing crossed a threshold, no profile change happens this time — the scorecard append from step 1 still happened regardless.
6. This skill does not report back to the user that a level changed unless they ask (e.g. via `learning-trend`, or by directly asking "have I improved?"). `ticket-coach` is not told whether a threshold was crossed either — it only hands off the scorecard and moves on, exactly like logging to `drift-log.jsonl`.

## FLOW E — Read scorecard history for another skill (e.g. learning-trend)

1. Read `scorecards.jsonl` in full.
2. Return all entries to the consuming skill, in the order they appear in the file (oldest first), with status `scorecards_found`. If the file doesn't exist yet or is empty, return status `no_scorecards_yet`.
3. The user is not informed of this read operation — it is transparent, like FLOW C.
4. This skill does not interpret or summarise the data for the consuming skill beyond returning it — presentation and trend analysis is `learning-trend`'s job, not this skill's.

## Observed evidence logging (for use by ticket-coach or other skills)

Any skill that detects a possible cognitive drift signal (e.g. the user repeatedly asks for "the code first" despite `concreteness` being marked low) may log it, but **must not write directly to `profile.json`**. Instead, it should record the observation in a separate evidence file:

```
~/.config/opencode/data/cognitive-profile/drift-log.jsonl
```

Each line is a JSON object: `{ "date": "...", "axis": "concreteness", "signal": "...", "source_skill": "ticket-coach" }`.

This skill (`cognitive-profile`) is solely responsible for reading `drift-log.jsonl`, deciding whether it crosses the evidence threshold (minimum 3 consistent signals out of the last 5 logged sessions) and internally triggering an `automatic_drift_threshold` reassessment — without the user needing to request it, but always confirming the result with them before applying it (showing qualitative statements just as in FLOW B; never applying silent changes without user confirmation).

By default, this skill ALWAYS confirms with the user before applying any score change, regardless of the trigger.

## Validation rules (summary — see schema.json for the formal detail)

- All `cognitive_profile` scores: range 1–5.
- All `cognitive_weights` values: range 0–1, and each opposite pair must sum to 1.0 (± 0.01 tolerance).
- `delta_max` per reassessment: ±0.5 over the previous score, never more.
- Never write a document that fails validation against `schema.json`.
- `additional_context` is an append-only array (previous entries are never deleted); if the user has nothing new to add, do not append an empty entry.

## Language

- The language of every interaction in this skill (questionnaire prompts, confirmations, reassessment statements) always follows the language of the user's most recent message in the current session.
- `background_context.is_english_native` and `response_language_variant` are used SOLELY to calibrate vocabulary complexity within whichever language the user is already using — never to decide which language to respond in.
- `response_language_variant` only ever holds short locale-style codes (e.g. `{ "en": "en-GB" }`), never prose instructions about tone or vocabulary. Any guidance about *how* to simplify vocabulary belongs to the `is_english_native` boolean and this skill's own behaviour — not to a free-text value inside `response_language_variant`.
